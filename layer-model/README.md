# Layer Model

Aggregate **nested columns** — lists, maps, structs — without ever materializing row objects.
Parquet stores only flat leaf columns and rebuilds nesting from Dremel's repetition and definition
levels; Hardwood's **layer model** hands you that structure already decoded, as a small stack of
*layers* above each leaf, so you can count and slice at columnar speed.

The headline this example makes concrete: a list's two schema groups, and a map's two schema
groups, each **fold into a single repeated layer** — so the very same offset-walking code
summarizes a `list` and a `map`.

This is the columnar counterpart to the Row API [`nested-data`](../nested-data) example; both read
the same bundled fixtures (no download).

## What you'll learn

- Read a leaf's layer stack with `getLayerCount()` / `getLayerKind(layer)` — each layer is either
  `REPEATED` (a list or map) or `STRUCT` (an optional struct).
- Slice a flat value array back into per-record lists/maps with `getLayerOffsets(layer)`, and tell
  a **null** parent from a **present-but-empty** one with `getLayerValidity(layer)`.
- See that a `list` (`LIST` group + inner `repeated` group) and a `map` (`MAP` group +
  `key_value` group) each collapse to **one** `REPEATED` layer — one routine handles both.
- Use `getLeafValidity()` to find null leaf values under a struct — and see that a struct adds its
  own `STRUCT` layer rather than folding away.


## The three-level encoding, briefly

Every leaf value Parquet stores carries two small integers:

- a **repetition level** `r` — at which level the value *continues* a repeated parent (`r = 0`
  starts a new record),
- a **definition level** `d` — how far down the optional/repeated path the value is actually
  *defined* (`d = max` means the leaf itself is present).

For the address book's `ownerPhoneNumbers.list.element` (max `r = 1`, max `d = 3`) the two records
encode as:

| `r` | `d` | meaning |
| --- | --- | --- |
| 0 | 3 | `555 123 4567` — new record, element present |
| 1 | 3 | `555 666 1337` — *same* list (r=1 reopens it), element present |
| 0 | 1 | new record, list is present but **empty** (d < max, so no value) |

The layer model turns those level streams into something you can index directly: the repetition
levels become **offsets**, and the definition levels become **validity**. The two schema groups a
`LIST` (or `MAP`) introduces only ever produce a single repetition level, which is why they fold
into one repeated layer.

## Run it

No setup beyond a JDK 21+. Pick either Maven or Docker.

**Maven** (from this folder, using the bundled Maven wrapper):

```shell
cd layer-model
./mvnw -q compile exec:java
```

**Docker:**

```shell
cd layer-model
docker compose run --rm --build layer-model
```

The sample files are bundled on the classpath and read straight into memory, so the example
works offline with nothing to download.

## Expected output

```
=== Repeated layers: list and map fold the same way ===
ownerPhoneNumbers.list.element: 2 phone numbers across 2 owners (most in one: 2; empty: 1; null: 0)
int_map.key_value.value: 4 settings across 3 users (most in one: 3; empty: 1; null: 0)

=== Struct layer: find contacts with no phone number ===
contacts.list.element.phoneNumber: layers [REPEATED > STRUCT]
  2 contacts, 1 with no phone number
```

`summarizeRepeated` is called once for a list and once for a map and behaves identically — that
sameness *is* the folding. The empty counts come from records whose offset span is zero
(`A. Nonymous`'s phone list; the third user's empty settings map).

## How it works

[`Main.java`](src/main/java/dev/hardwood/examples/layermodel/Main.java) has two helpers;
[`Datasets.java`](src/main/java/dev/hardwood/examples/layermodel/Datasets.java) reads the bundled
fixtures from the classpath into an in-memory `InputFile`.

- **One repeated layer per list/map.** `summarizeRepeated` opens a leaf, asserts a single
  `REPEATED` layer, then reads `getLayerOffsets(0)`. For record `r`, `offsets[r+1] - offsets[r]` is
  its item count; a zero-width span is an empty parent. `getLayerValidity(0)` separates a null
  parent (the whole list/map absent) from a present-but-empty one.
- **Offsets index the flat values.** The leaf values are stored once, contiguously; the offsets are
  all you need to walk them back into per-record lists — no per-row objects allocated.
- **Structs don't fold.** `contacts.list.element.phoneNumber` reports layers `REPEATED > STRUCT`:
  the list folds to one repeated layer, but the optional struct contributes its own `STRUCT` layer
  (validity, no offsets). `getLeafValidity()` then marks the null leaf values — here, the one
  contact with no phone number, the same answer the Row API example computes record by record.
